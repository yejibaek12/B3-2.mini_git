# Mini Git

Mini Git은 커밋 그래프(DAG), 브랜치 제어, 위상 정렬 기반 로그 출력, 고속 검색 색인, 최단 경로 탐색 등 Git의 핵심 메커니즘을 메모리 상에서 시뮬레이션하는 명령줄 인터페이스(CLI) 프로그램입니다.

---

## 1. Quick Start

### 실행 방법
진입점 파일인 `main.py`가 있는 디렉터리에서 아래의 명령어를 실행하여 대화형 CLI 콘솔을 작동시킵니다.
```bash
python main.py
```

## 2. 개발 제약 사항 및 범위

### 기술 제약 및 라이브러리 제한
* **개발 언어**: Python 3.10 이상 권장.
* **라이브러리 제한**: 그래프 표현/탐색 전용 서드파티 라이브러리 사용을 일절 금지합니다.
* **내장 기능 제한**: 파이썬 내장 정렬 API인 `sorted()`, `list.sort()` 등의 사용이 전면 불가능하며 직접 정렬 로직을 짜서 적용해야 합니다.

### 구조 및 품질 기준 (코드 스타일)
* **알고리즘 로직 분리**: 탐색, 정렬, 인덱싱 등의 핵심 알고리즘 로직은 독립된 함수 또는 클래스로 명확히 분리하여 구현합니다.
* **문서화**: 주요 함수와 클래스에는 역할과 동작 방식을 설명하는 주석 또는 docstring을 상세히 작성합니다.

### 기능 구현 범위 (Scope)
* **커밋 메타데이터 중심**: 실제 파일 내용 추적(File Content Tracking)은 구현하지 않으며, 커밋 메타데이터만을 관리 대상으로 삼습니다.
* **네트워크 통신 배제**: 분산 환경 처리를 위한 네트워크 통신 기능은 구현하지 않습니다.
* **인메모리 동작**: 데이터 영속성(파일 저장 등)은 구현하지 않으며, 메모리 상에서의 동작으로 충분합니다.

## 3. 프로젝트 구조 및 소스 코드 구성

이 프로젝트는 로컬 설계 문서를 제외하고 배포에 필요한 모든 기능적 핵심 로직이 단 하나의 파일인 `main.py`에 통합적으로 구현되어 있어 간편하게 실행할 수 있습니다.

* **`main.py` 내부 소스 코드 구성**
  * **SECTION 1. UTILS & HELPER ALGORITHMS**: 정렬 제약을 해결하기 위해 직접 구현한 Stable $O(N \log N)$ 병합 정렬 (`merge_sort`). 추가 임시 배열 메모리 할당으로 인해 $O(N)$ 공간 복잡도를 가집니다.
  * **SECTION 2. DATA MODELS**: Git의 개별 커밋 노드를 설계한 `CommitNode` 클래스.
  * **SECTION 3. CORE ENGINE (Repository)**
    * **3-1. Engine & Hash Generator**: 저장소 상태 및 순차 해시 생성기.
    * **3-1-Helper. Reusable Graph Builder**: `path`, `ancestors`, `log` 등에서 중복 사용되는 그래프 생성 연산을 통합 관리하는 공통 헬퍼 메서드 `_build_adjacency_list`.
    * **3-2. Command Handlers**: 저장소 초기화 및 `INIT`, `BRANCH`, `SWITCH`, `COMMIT` 등의 브랜치 제어 메서드.
    * **3-3. Search Engine**: 딕셔너리를 활용한 고속 인메모리 역색인 검색기.
    * **3-4/3-5. Graph Algorithms**: 위상 정렬 기반 `LOG` 조회, 무방향 BFS 탐색을 통한 `PATH` 추적, DFS 기반의 `ANCESTORS` 조상 조회 및 브랜치 `MERGE`.
  * **SECTION 4. INTERACTIVE CONSOLE (REPL)**: 예외 처리와 명령어 토크나이저(`shlex`)가 적용된 `mini-git>` 대화형 셸 콘솔 메인 루프.

## 4. 주요 기능

### 1. 저장소 및 브랜치 관리
* 저장소를 초기화하고 기본 브랜치(`main`)와 현재 사용자를 설정합니다.
* 새로운 브랜치를 생성하고 자유롭게 브랜치를 전환합니다.
* 메시지, 작성자, 타임스탬프, 부모 해시 등의 메타데이터를 담은 커밋을 생성합니다.

### 2. 커밋 로그 및 그래프 탐색
* 부모 커밋이 자식 커밋보다 항상 먼저 나타나도록 Kahn 알고리즘 기반 위상 정렬로 전체 로그를 출력합니다. (동일 타임스탬프 발생 시 커밋 해시 사전순으로 결정적인 출력 순서를 가집니다.)
* 무방향 간선 모델을 기준으로 임의의 두 커밋 간 최단 경로를 탐색합니다.
* 특정 커밋으로부터 도달 가능한 모든 조상 커밋 목록을 출력합니다.

### 3. 검색 및 정렬
* 메시지 내 단어 키워드 혹은 작성자 이름 기반의 역색인(Inverted Index)을 구축하여 검색 요청을 고속으로 처리합니다.
* 직접 구현한 병합 정렬 알고리즘을 사용하여 날짜 또는 작성자별로 로그 정렬 조회를 제공합니다.

### 4. 대화형 CLI 인터페이스 (REPL)
* **입력/요청**: 프로그램이 실행되면 `mini-git> ` 프롬프트가 노출되며 사용자의 입력을 대기하고, 이 프롬프트 상에서 명령을 반복하여 입력할 수 있습니다.
* **출력/화면**: 입력받은 명령을 토큰 단위로 파싱하고 실행하여, 그 처리 결과를 화면에 노출하는 일련의 입출력 루프가 반복적으로 수행됩니다.
* **종료**: `EXIT` 또는 `QUIT` 명령을 입력하면 REPL 루프를 종료하고 프로그램을 안전하게 마칩니다.

## 5. 명령어 규격 및 레퍼런스

### 명령어 입력 표준 규칙
* **대소문자 구분 없음**: 명령어 키워드는 대소문자를 구분하지 않습니다 (예: `INIT`와 `init` 동일하게 동작).
* **공백 포함 인수**: 공백이 포함된 문자열 인수(사용자명/커밋 메시지/검색 키워드)는 반드시 큰따옴표로 감싸서 입력해야 합니다 (예: `COMMIT "Add login feature"`).
* **에러 메시지 규격화**
  * 인수 누락 또는 옵션 형식 오류: `Invalid args`
  * 존재하지 않는 브랜치 대상 요청: `Unknown branch: <name>`
  * 존재하지 않는 커밋 해시 대상 요청: `Unknown commit: <hash>`
  * 저장소 미초기화 상태에서 명령어 입력: `Repository not initialized`

### 명령어 상세 레퍼런스

#### 저장소 및 브랜치 제어
* `INIT <user_name>`
  * 저장소를 초기화합니다. 기본 브랜치인 `main`을 생성하여 활성 브랜치(`HEAD`)로 삼고 작성자를 `<user_name>`으로 고정합니다.
  * **예외 및 제약사항**: 이미 초기화된 상태에서 `INIT`을 다시 호출하면, 기존 메모리에 적재된 커밋 그래프, 브랜치 목록, 역색인 및 해시 카운터 데이터가 전부 유실(포맷)되고 새 저장소로 완전히 재시작됩니다.
* `BRANCH <branch_name>`
  * 현재 활성 커밋(`HEAD`) 위치에 지정한 이름의 새 브랜치 포인터를 생성합니다.
  * **예외 및 제약사항**: 이미 존재하는 이름으로 브랜치를 중복 생성하려고 시도할 경우, 생성 작업을 거부하고 `Branch already exists: <branch_name>`을 출력합니다.
* `SWITCH <branch_name>`
  * 활성 브랜치(`HEAD`)를 지정한 이름의 브랜치로 전환합니다.
* `COMMIT <message>`
  * 현재 `HEAD` 커밋을 부모 노드로 삼는 새 커밋 객체를 기록하고 소속 브랜치의 팁 포인터를 새 커밋 해시값으로 업데이트합니다.
  * 작성자 및 메시지 단어에 대한 역색인을 즉시 업데이트합니다.

#### 로그 및 그래프 분석
* `LOG`
  * 위상 정렬 순서(부모 커밋이 자식 커밋보다 앞서서 출력되는 형태)에 맞게 전체 커밋 로그를 보여줍니다.
  * **결정적 위상 정렬 규칙**: 다중 브랜치나 병합 커밋 등으로 인해 큐에 동시에 들어오는 동일 수준(동일 타임스탬프)의 커밋 노드들은 커밋 해시의 알파벳 사전순으로 오름차순 정렬하여 출력함으로써 항상 동일한 실행 결과를 보장합니다.
* `LOG --sort-by=date|author`
  * 지정한 기준대로 전체 커밋 로그를 정렬하여 보여줍니다.
    * `date`: 타임스탬프 시간 순으로 정렬합니다.
    * `author`: 작성자 사전순으로 정렬합니다.
  * **유효성 검사**: `--sort-by` 뒤에 잘못된 파라미터가 입력될 경우 `Invalid args`를 출력합니다.
* `PATH <commit1> <commit2>`
  * 두 커밋을 연결하는 최단 경로(최소 간선 수)를 탐색합니다. (무방향 그래프 간선 간주)
  * **결정적 동률 경로 처리**: 여러 개의 최단 경로가 요건을 충족한다면, 경로 전체를 화살표(` -> `)로 연결한 전체 경로 문자열을 사전순으로 비교하여 가장 앞에 위치하는 경로를 최종 선택합니다.
    * *예시*: 최단 경로로 `a1 -> b1 -> c1`과 `a1 -> b2 -> c1`이 동률로 존재할 때, 문자열 비교 시 `'a1 -> b1 -> c1'`이 `'a1 -> b2 -> c1'`보다 사전순으로 앞서므로 `Path: a1 -> b1 -> c1`을 출력합니다.
  * 경로가 아예 없는 경우 `No path`를 출력합니다.
* `ANCESTORS <commit_hash>`
  * 지정한 커밋에서 도달 가능한 모든 조상 커밋들을 빠짐없이 추적하여 출력합니다.
  * **정렬 규칙**: 수집된 조상 목록은 타임스탬프 시간 순(오름차순) 및 해시 사전순으로 정렬하여 일관되게 출력합니다. 조상이 없는 경우 `Ancestors: None`을 출력합니다.

#### 인메모리 검색 및 기타 제어
* `SEARCH <keyword>`
  * 메시지에 키워드가 포함된 커밋들을 역색인을 이용하여 고속 검색 후 출력합니다. (대소문자 무시 단어 기준)
* `SEARCH --author=<name>`
  * 해당 저자가 생성한 커밋 목록을 역색인을 이용해 고속 검색 후 출력합니다.
* `EXIT` 또는 `QUIT`
  * 대화형 CLI 콘솔(REPL) 루프를 탈출하여 프로그램을 안전하게 종료합니다. (매개변수 동반 입력 시 `Invalid args` 에러 출력)

## 6. 핵심 설계 원칙 및 메커니즘

### 커밋 그래프의 DAG(Directed Acyclic Graph) 지향성
Git은 역사적 흐름을 보존하기 위해 방향성 비순환 그래프(DAG) 아키텍처를 가집니다.
* **사이클 방지(Acyclic)**: 커밋 이력에서 과거 커밋이 미래에 새로 만들어질 커밋을 부모로 참조할 수 없습니다. 만약 사이클이 발생한다면 이력 추적 루프가 무한 루프에 빠지게 되며, 버전 관리의 타임라인 인과관계가 무너집니다.
* **이력 불변성(Immutability)**: 생성된 커밋 노드는 고유한 메타데이터(부모 해시, 메시지, 시간)를 품으며, 한 번 작성된 과거 이력은 신규 커밋에 의해 변경되거나 오염되지 않고 불변 상태로 유지됩니다.

### 순차 해시 생성 메커니즘
* 현재 구현체는 테스트 및 디버깅의 편의성을 위해 순차 카운터를 기반으로 한 고유 패턴 해시(`a1b2c3`, `d4e5f6` 등)를 생성합니다.
* **충돌 방지 보장**: 동일 세션 내에서 카운터가 단조 증가하므로 생성되는 해시값은 절대 중복되지 않으며 충돌의 여지가 존재하지 않습니다.

## 7. 대규모 확장성 분석 및 설계 제안

### 인메모리 아키텍처의 한계와 개선 방안
현재 프로그램은 오직 메모리상에서만 데이터 구조(`dict`, `list`)를 유지하므로 만 개 이상의 커밋이 누적될 경우 다음과 같은 문제가 생길 수 있습니다.
* **메모리 오버헤드**: 파이썬 객체로 모든 노드를 유지하므로 RAM 사용량이 가파르게 상승합니다.
* **영속성 부재**: 프로세스 종료 시 전체 작업 내역이 소실됩니다.
* **해결 제안**: 영속성과 효율적인 메모리 페이징을 위해 파일 기반 경량 엔진인 **SQLite**나 **LevelDB**를 스토리지 계층으로 분리하여 마이그레이션해야 합니다.

### 역색인 자료구조의 최적화
* **현재 구현**: 역색인의 밸류 타입으로 `list`를 사용합니다. 이 경우 동일 커밋 해시가 중복 등록되거나 특정 커밋을 색인에서 제거할 때 리스트 순회 비용 $O(N)$이 발생합니다.
* **개선 제안**: 중복을 원천 차단하고 $O(1)$의 추가/삭제 성능을 보장하기 위해 역색인 구조를 `dict[term, set[commit_hash]]` 형태로 이주할 것을 권장합니다.

### 역색인 업데이트의 트랜잭션 부재
* **현재 구현**: 커밋 작성 시 메시지 인덱싱과 저자 인덱싱이 연속적으로 딕셔너리에 추가됩니다. 도중에 예외가 발생하더라도 롤백할 수 있는 트랜잭션 장치가 없습니다.
* **분석 및 제안**: 현재 시스템은 단일 스레드 인메모리 동기식 동작이므로 DB 접속 장애나 파일 쓰기 실패 같은 런타임 부분 실패가 일어나지 않습니다. 다만 멀티스레드나 외부 영속성 계층 도입 시에는 전체 쓰기가 성공하거나 전혀 반영되지 않도록 하는 **원자적 트랜잭션 및 락(Lock) 메커니즘**의 추가 도입이 요구됩니다.

### 보안성 확보 및 무작위 해시 대체
* **현재 구현**: 순차 카운터 기반 해시는 추측하기 너무 쉬워 실프로덕션 환경에서 보안 취약점이 됩니다.
* **개선 제안**: 분산 환경에서 여러 작업자가 동시에 커밋을 작성할 때의 해시 고유성 확보 및 변조 방지를 위해, Git 순정 방식과 같이 커밋 메타데이터 전체를 솔트로 삼아 해싱하는 **SHA-1** 혹은 **SHA-256** 기반 암호학적 무작위 해시 생성 로직으로 대체해야 합니다.
